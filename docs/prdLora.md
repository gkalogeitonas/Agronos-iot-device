### 📌 PRD: Edge Node Firmware (Agronos-iot-device)

**Feature:** LoRaWAN-Compliant E2EE, Binary Serialization & Hardware Wear Leveling
**Context:** C++ / PlatformIO σε ESP32 / Ενσωμάτωση με LoRa Radio

#### 1. Στόχος (Objective)

Η ανάπτυξη του υποσυστήματος επικοινωνίας LoRa στο firmware του ESP32, το οποίο θα συλλέγει μετρήσεις από τους αισθητήρες, θα τις σειριοποιεί σε αυστηρό δυαδικό format (binary serialization), θα κατασκευάζει το ντετερμινιστικό Nonce και θα κρυπτογραφεί το ωφέλιμο φορτίο (payload) μέσω υλικής επιτάχυνσης (AES-128-CTR), πριν τη μετάδοση στο LoRa Gateway.

#### 2. Αρχιτεκτονικές Απαιτήσεις (Architectural Requirements)

**2.1. Δυαδική Σειριοποίηση Δεδομένων (Binary Payload Serialization)**

* **Λογική:** Εγκατάλειψη του JSON. Το payload πρέπει να έχει μεταβλητό μήκος **$N \times 6$ bytes**, όπου $N$ είναι ο αριθμός των συνδεδεμένων αισθητήρων.
* **Υλοποίηση ανά Αισθητήρα (Chunking):**
* **Bytes 0-3:** Εξαγωγή των πρώτων 4 χαρακτήρων (ASCII) από το UUID του αισθητήρα.
* **Bytes 4-5:** Η float τιμή της μέτρησης πολλαπλασιάζεται επί 100, μετατρέπεται σε ακέραιο (`int16_t`) και τοποθετείται στη μνήμη σε μορφή **Little-Endian (LE)**.


* *Σημείωση Αρχιτεκτονικής:* Το ESP32 χρησιμοποιεί εγγενώς Little-Endian αρχιτεκτονική μνήμης, οπότε ένα απλό `memcpy` ή casting θα λειτουργήσει ιδανικά χωρίς ανάγκη για byte-swapping.

**2.2. Διαχείριση Frame Counter & Προστασία Μνήμης Flash**

* **Λογική:** Διατήρηση του μονοτονικά αυξανόμενου μετρητή (`fcnt`) για την αποτροπή Replay Attacks, προστατεύοντας παράλληλα τη μνήμη NVS από φθορά (Wear Leveling).
* **Υλοποίηση (Υβριδική Μνήμη):**
* Αποθήκευση του τρέχοντος `fcnt` στη μνήμη RTC του ESP32 (π.χ., με χρήση `RTC_DATA_ATTR`) ώστε να επιβιώνει από τον κύκλο του Deep Sleep.
* Εφαρμογή λογικής "Lazy Save": Η αποθήκευση του μετρητή στη μόνιμη μνήμη NVS θα γίνεται μόνο κάθε π.χ., 100 μεταδόσεις.
* Σε περίπτωση κρύας εκκίνησης (Cold Boot / Power Loss), ο κόμβος θα διαβάζει τον τελευταίο μετρητή από την NVS, θα προσθέτει ένα ασφαλές άλμα (π.χ., +100) και θα συνεχίζει από εκεί. Αυτό το άλμα υποστηρίζεται ήδη στο backend χάρη στο `MAX_FCNT_GAP = 10.000` που υλοποίησες.



**2.3. Ντετερμινιστική Κατασκευή Nonce**

* **Λογική:** Δημιουργία του 16-byte Nonce στη μνήμη RAM του ESP32 πριν από κάθε κρυπτογράφηση, χωρίς αυτό να μεταδίδεται ολόκληρο στον αέρα.
* **Υλοποίηση:** Το array των 16 bytes πρέπει να δομηθεί ακριβώς όπως το περιμένει η `LoRaCryptoService`:
* **Bytes 0-3:** Το Device ID του κόμβου σε μορφή Little-Endian (LE).
* **Bytes 4-7:** Ο τρέχων Frame Counter (`fcnt`) σε μορφή Little-Endian (LE).
* **Bytes 8-15:** Γεμισμένα με μηδενικά (0x00).



**2.4. Υλικό-Επιταχυνόμενη Κρυπτογράφηση (AES-128-CTR)**

* **Λογική:** Κρυπτογράφηση του serialized binary payload με τη χρήση του κρυπτογραφικού συνεπεξεργαστή του ESP32.
* **Υλοποίηση:** Χρήση της βιβλιοθήκης `mbedtls` (ενσωματωμένη στο PlatformIO/ESP-IDF). Το 16-byte `lora_aes_key` (που γνωρίζει το backend) και το κατασκευασμένο Nonce θα τροφοδοτήσουν τη λειτουργία AES σε Counter Mode (CTR). Η κρυπτογράφηση θα παράγει ένα ciphertext με ακριβώς το ίδιο μέγεθος με το αρχικό plaintext ($N \times 6$ bytes), χωρίς καθόλου padding.

**2.5. Δομή Τελικού Πακέτου Μετάδοσης (LoRa Transmission)**

* **Λογική:** Αποστολή των ελάχιστων απαραίτητων δεδομένων στο LoRa Gateway.
* **Υλοποίηση:** Ο κόμβος θα εκπέμψει μέσω του LoRa module (π.χ., SX1276/RFM95) ένα πακέτο που θα περιέχει:
1. Το Device ID (4 bytes)
2. Τον Frame Counter (4 bytes)
3. Το AES-128-CTR Ciphertext ($N \times 6$ bytes).
*Το Gateway έπειτα θα αναλάβει να τα πακετάρει στο JSON wrapper (που περιλαμβάνει τα RSSI/SNR) και να τα στείλει στον EMQX Broker.*
